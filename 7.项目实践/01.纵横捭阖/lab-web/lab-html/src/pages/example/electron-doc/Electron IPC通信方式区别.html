Electron IPC 通信方式详解

invoke、send 和 sendSync 的全面对比与示例

三种 IPC 方式核心区别

特性 ipcRenderer.invoke() ipcRenderer.send() ipcRenderer.sendSync()
通信模式 异步请求-响应 异步单向消息 同步请求-响应
返回值 Promise (async/await) 无返回值 直接返回值
阻塞性 非阻塞 非阻塞 阻塞渲染进程
错误处理 Promise.catch() 或 try-catch 事件监听错误 try-catch
主进程监听 ipcMain.handle() ipcMain.on() ipcMain.on()
主进程响应 return 或 Promise.resolve event.reply() event.returnValue
适用场景 需要响应的异步操作 单向通知或长时间操作 快速同步操作
代码复杂度 低 (推荐) 中 低 (谨慎使用)

使用场景详解

场景1：用户数据获取 - 使用 invoke（推荐）

// 渲染进程 - 简洁的异步代码 async function loadUserProfile(userId) { try { showLoading(); const userData = await ipcRenderer.invoke('get-user-profile', userId); displayUserProfile(userData); } catch (error) { showError('加载用户资料失败: ' + error.message); } finally { hideLoading(); } } // 主进程 - 清晰的处理器 ipcMain.handle('get-user-profile', async (event, userId) => { // 参数验证 if (!userId) { throw new Error('用户ID不能为空'); } // 异步操作 const user = await database.getUser(userId); const profile = await fetchUserProfile(user.profileUrl); return { ...user, profile }; });

优势：代码简洁，错误处理方便，非阻塞UI

场景2：文件上传进度 - 使用 send

// 渲染进程 - 进度监听 function uploadFile(file) { // 发送上传请求 ipcRenderer.send('start-file-upload', file); // 监听进度更新 ipcRenderer.on('upload-progress', (event, progress) => { updateProgressBar(progress.percentage); }); // 监听完成 ipcRenderer.once('upload-complete', (event, result) => { showSuccess('上传完成'); handleUploadResult(result); }); // 监听错误 ipcRenderer.once('upload-error', (event, error) => { showError('上传失败: ' + error); }); } // 主进程 - 长时间操作 ipcMain.on('start-file-upload', async (event, file) => { try { const uploader = new FileUploader(file); uploader.on('progress', (progress) => { event.reply('upload-progress', progress); }); const result = await uploader.upload(); event.reply('upload-complete', result); } catch (error) { event.reply('upload-error', error.message); } });

优势：适合长时间操作，支持进度反馈，多事件通信

场景3：应用配置读取 - 使用 sendSync

// 渲染进程 - 同步获取配置 function initializeApp() { try { // 快速同步获取配置 const config = ipcRenderer.sendSync('get-app-config'); // 立即使用配置 applyConfiguration(config); setupAppWithConfig(config); } catch (error) { // 使用默认配置 useDefaultConfiguration(); console.warn('使用默认配置: ' + error.message); } } // 主进程 - 快速同步响应 ipcMain.on('get-app-config', (event) => { try { // 从文件同步读取配置 const configPath = getConfigPath(); if (!fs.existsSync(configPath)) { throw new Error('配置文件不存在'); } const configData = fs.readFileSync(configPath, 'utf8'); const config = JSON.parse(configData); event.returnValue = config; } catch (error) { // 发送错误信息 event.returnValue = { error: error.message }; } });

优势：立即获取结果，代码简单，适合快速操作

注意：仅适用于耗时极短的操作（<100ms）

最佳实践指南

1. 优先使用 invoke/handle

  • 代码更简洁，支持 async/await
  • 内置错误处理机制
  • 不会阻塞渲染进程
  • Electron 官方推荐的新模式

2. send/on 适用于复杂场景

  • 需要进度更新的长时间操作
  • 多个事件需要通信的场景
  • 单向通知不需要响应的情况

3. 谨慎使用 sendSync

  • 仅用于耗时极短的操作（<100ms）
  • 避免在渲染进程繁忙时使用
  • 始终使用 try-catch 包装
  • 考虑是否有异步替代方案

4. 错误处理规范

// invoke - 使用 try-catch try { const result = await ipcRenderer.invoke('some-action'); } catch (error) { // 处理错误 } // send - 监听错误事件 ipcRenderer.send('some-action'); ipcRenderer.once('action-error', (event, error) => { // 处理错误 }); // sendSync - 同步 try-catch try { const result = ipcRenderer.sendSync('some-action'); } catch (error) { // 处理错误 }

5. 性能优化建议

  • 批量处理多个IPC调用
  • 避免频繁的小消息通信
  • 使用数据压缩对于大数据传输
  • 考虑使用Web Workers处理CPU密集型任务
